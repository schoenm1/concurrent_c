\section{Speicherverwaltung -- Buddy System}
Für die Verwaltung des Shared Memory (shm) bedarf es einer Logik, um die verschiedenen Adressen im Shared Memory richtig ansprechen zu können. Zusätzlich muss sichergestellt werden, dass kein File in das shm geschrieben wird, dass länger ist als der freie Speicherplatz, bevor das nächste File kommt.\\
Es gibt viele dokumentierte Speicherverwaltungen. Nach längerer Recherche wurde entschieden, dass der Speicher mit dem Buddy-System verwaltet werden soll.
Die Suche im Internet nach einer vorhandenen Library für die Speicherverwaltung mit dem Buddy-System blieb leider erfolglos. Also blieb nichts anderes übrig, als das Buddy-System von Grund auf selber zu gestalten und zu implementieren.\\
Dabei wurden sehr viele Fragen aufgeworfen, welche Schrittweise erarbeitet wurden.

Die nächsten Unterkapitel zeigen einige aufgeworfene Fragen, welche gelöst werden mussten.

\subsection{Wie wird der optimale Block für ein neues File im Shared Memory gefunden?}

Hierzu wurde die Funktion find\_shm\_place(\dots) erstellt.\\
Diese Funktion beginnt beim ersten Eintrag des Structs shm\_ctr\_struct (siehe Listing \ref{shmctrstruct}) und sucht über alle vorhanden Blöcke (über den next-Pointer) einen optimalen Block. Optimal bedeutet, dass er grösser oder gleich der Grösse der neu zu erstellenden Dokumentes  sein muss, aber nicht grösser als das Doppelte. Wäre er grösser als das Doppelte, wäre dies Speicherplatzverschwendnung. Zusätzlich muss der Block frei sein (isfree = TRUE).

\subsection{Gibt es nur zu grosse Blöcke, wie werden die aufgetrennt?}

Für die Aufteilung der Blöcke wurde die Funktion devide(\dots) implementiert.\\
Diese beginnt beim ersten Block und arbeitet sich (über den next-Pointer) nach hinten. Beim ersten gefundenen freien Block, wird nun die Block-Size halbiert.
Es wird ein neuer Block erzeugt und die Verlinkungen (next, previous, Pointer auf Filename und Filedata sowie isFree und size) werden dem bestehenden und neuen Block gesetzt, so dass die Linked-List wieder komplett vorhanden ist.\\
Ist die Blockgrösse die gewünschte Grösse, findet ein return = TRUE statt. Ansonsten wird die Funktion  rekursiv aufgerufen, bis die Blockgrösse genügend klein ist. Dann erfolgt der return = TRUE.\\
Ein Screenshot der Funktion devide() ist im Anhang \ref{fig:shm-2} zu finden. Ebenfalls im Anhang \ref{fig:shm-1} und \ref{fig:shm-3} ist das Shared Memory vor und nach dem Einfügen eines neuen Files zu sehen.



Das Buddy-System gibt vor, dass die Blockgrössen aus 2er Potenzen gebildet werden. Also 2, 4, 8, 16, 32 \dots \\

\newpage
Das Vorgehen beim Aufteilen der Blöcke ist folgendermassen:
\begin{enumerate}
\item Durch den Aufruf der Funktion wird die gewünschte Grösse des Blockes mitgegeben. Da diese Funktion nur aufgerufen wird, wenn der Block noch nicht existiert, muss nicht mehr geprüft werden, ob es bereits einen optimalen Block gibt.\\
Dessen Grösse ist in 2er Potenzen (in unserem Beispiel unten 16384)
\item Beginne beim 1. SHM-Block. Ist dieser grösser als 16384 und frei? Wenn ja, gehe zu Punkt 3, sonst wiederhole diesen Punkt bis Bedingung erfüllt ist.
\item Wurde ein Block zum Teilen gefunden, setzte die neuen Parameter (hier nur Auszugsweise dargestellt).\\
Der Quellcode ist im Listing \ref{listing:devide} im Anhang zu finden.
\begin{itemize}
\item erstelle eine neue Instanz des Struktes shm\_ctr\_struct
\item Halbiere die Grösse des momentanen Blockes mit\\
 shm\_ctr->shm\_size = (shm\_ctr->shm\_size) / 2;
\item setzte next des aktuellen Struktes auf das neu Erstellte
\item setze den Vorgänger des neuen Struktes auf das akutelle Strukt mit\\
nextsshm->prev=shm\_ctr
\item ... (für die komplette Implementierung siehe \ref{listing:devide} im Anhang)
\end{itemize}
\end{enumerate}

\section{Locks}

Wie im Kapitel \ref{sec:Das Projekt} erwähnt, ist ein global Lock nicht erlaubt.\\
Für die Umsetzung des Locks wurde schlussendlich kein mutex gewählt wie anfangs angedacht war. Das Problem beim mutex ist, dass ein lesender Client das ganze File ebenfalls sperrt für weitere Lesezugriffe. Dies soll jedoch nicht der Fall sein.\\
Aus diesem Grund wurde wurde auf \flqq pthread\_rwlock\_t\frqq\ zurückgegriffen.

Die Implementation des Locks wurde gemäss Tabelle im Kapitel \ref{subsec:completeSystem} vorgenommen.\\
Da das Kontroll-Strukt für das Shared-Memory bereits vorhanden war, konnte\\
 \flqq pthread\_rwlock\_t\frqq\ ohne Probleme eingefügt werden.\\
Soll nun ein ein File gelockt werden, kann das elegant gelöst werden:
\begin{itemize}
\item Wenn das File gelesen werden möchte, muss zwingend die Adresse des enstprechenden Strukt bereits vorhanden sein = struct shm\_ctr\_struct *shm\_ctr
\item Nun kann ein ReadLock über pthread\_rwlock\_rdlock(\&(shm\_ctr->rwlockFile)); gemacht werden.
\end{itemize}

Eine kurze Übersicht über die Impelementation des Locks gibt das Kapitel \ref{sec:Server-Befehle}, welches die einzelnen CRUD Befehle aufführt.

\section{CREATE}
\label{subsec:CREATE}

Das vereinfachte Vorgehen beim CREATE ist folgendermassen:
\begin{enumerate}
\item Überprüfe, ob Filename und Filecontent mitgegeben wurde. Fehlt das eine oder andere, sende dem Client direkt zurück, dass File nicht erstellt werden kann.
\item Wurde Filename und Fileconent mitgegeben, überprüfe ob das File bereits existiert. Wenn ja: return \flqq File already exist\frqq. Wenn nein, Schritt 3
\item Suche einen guten Platz im Shared Memory. Aufgrund der Implementierung eines Buddy-Systems muss der SHM-Block grösser sein als das einzufügende File, aber kleiner als das doppelte. Diese Überprüfung wird von der Funktion \flqq find\_shm\_place(\dots)\frqq\ übernommen.
\item Wenn kein Platz gefunden: return -1, wenn Platz vorhanden: weiter mit Schritt 5
\item setze Flag für den gefunden SHM-Block = not free
\item initialisiere den RW Lock im Strukt shm\_ctr\_struct
\item setze den Write-Lock
\item schreibe Filename und Fileinhalt
\item gib den Write Lock wieder frei
\item return: \flqq File xyz sucessfully created\frqq
\end{enumerate}
\textbf{Beispiel für gültige Eingaben von CREATE:}\\
--CREATE test.java Dies ist der Inhalt vom test.java file

\textbf{Beispiele für ungültige Eingaben von CREATE:}\\
--CREATE

Wenn kein Filename mitgegeben wird, sendet der Server folgende Nachricht an den Client:\\
\flqq Can not create a new file. Filename is missing.\frqq

--CREATE filename.txt

Wenn ein Filename, aber kein content mitgegeben wurde, sendet der Server folgende Nachricht an den Client: \flqq No file content set. File not created\frqq

Der Programmcode für das Handling von CREATE ist im Listing \ref{listing:CREATE} in Anhang \ref{Asubsec:CREATE} zu finden.  Die Einschränkung zu CREATE ist im Kapitel \refTC{Einschränkung_CREATE_UPDATE} zu finden.

\section{READ}
\label{subsec:READ}

Das Vorgehen bei einem READ Befehl ist folgendermassen:
\begin{enumerate}
\item Überprüfe ob ein Filename mitgegeben ist. Wenn nicht, sende an Client, dass nicht nach einem leeren Filenamen gesucht werden kann. Ist der Filename vorhanden, gehe zum nächsten Schritt.
\item Überprüfe im ersten SHM-Block, ob der Filename mit dem gesuchten Name übereinstimmt.
\item Gibt es einen Treffer:
\begin{itemize}
\item setze den Read-Lock
\item kopieren den Fileinhalt in einen temporären Char Pointer
\item gib den Read-Lock wieder frei
\item Gib als return Wert den content des Files
\end{itemize}
\item Wenn der Filename nicht übereinstimmt, gehe zum nächsten Block im SHM.
\item Wiederhole Punkt 4 so lange, bis ein Treffer vorhanden ist oder bis das Ende des SHM erreicht ist. Ist ein Treffer vorhanden, gehe zu Punkt 3.
\item Wurde das Ende des SHM erreicht und keine Übereinstimmung gefunden, sende \flqq File not found\frqq\ an den Client
\end{enumerate}


\textbf{Beispiel für gültige Eingaben von READ:}\\
--READ filename.txt

Als Antwort des Servers gibt es zwei Varianten. Entweder \flqq File not found\frqq\ oder der Inhalt des Files wird übermittelt.

\textbf{Beispiele für ungültige Eingaben von READ:}\\
--READ\\
Wenn kein Filename mitgegeben wird, antwortet der Server mit der Mitteilung: \flqq Cannot read an empty filename\frqq.

Der Programmcode für das Handling von READ ist im Listing \ref{listing:READ} in Anhang \ref{Asubsec:READ} zu finden.

\section{UPDATE}
\label{subsec:UPDATE}

Die Funktion für ein UPDATE eines Files wurde in zwei Teilschritten implementiert. Der erste Teil löscht das komplette File, der Zweite erstellt es mit dem neuen content wieder. Das Problem beim direkten Update eine Files könnte sein, dass der content grösser wird als der SHM-Block ist. Anstelle dies zu überprüfen, wurde global ein DELETE und CREATE implementiert.
\begin{enumerate}
\item Überprüfe ob ein Filename mitgegeben ist. Wenn nicht, sende an Client, dass ein File ohne Namen nicht geändert werden kann.
\item Überprüfe ob ein neuer Filecontent mitgegeben wurde. Wenn nicht, sende an den Client, dass der Content nicht leer sein darf.
\item Überprüfe, ob das File exisiert. Wenn nicht, sende an den Client, dass es nicht existiert.
\item Überprüfe im ersten SHM-Block, ob der Filename mit dem gesuchten Name übereinstimmt.
\item Gibt es einen Treffer:
\begin{itemize}
\item rufe Funktion deleteFile(\dots) auf - siehe Kapitel \ref{subsec:DELETE}
\item rufe Funktion createNewfile(\dots) auf - siehe  Kapitel \ref{subsec:CREATE}
\item sende dem Client die Nachricht, dass das File aktualisiert wurde.
\end{itemize}
\item Gibt es keinen Treffer, gehe zum nächsten SHM-Block und überprüfe, ob der Namen übereinstimmt. Wiederhole so lange, bis der Filename gefunden wurde. Bei einem Treffer, gehe zu Schritt 5
\end{enumerate}

\textbf{Beispiel für gültige Eingaben von UPDATE:}\\
--UPDATE test.txt dies ist der neue content

Als Antwort des Servers gibt es zwei Varianten. Entweder \flqq File with the name xyz does not exist!\frqq \ oder \flqq File with name xyz was successfully udpated.\frqq

\textbf{Beispiele für ungültige Eingaben von UPDATE:}\\
--UPDATE\\
--UPDATE test.txt

Wenn kein Filename mitgegeben wird, antwortet der Server mit der Mitteilung: \flqq Cannot update a file without a name\frqq.

Der Programmcode für das Handling von UPDATE ist im Listing \ref{listing:UPDATE} in Anhang \ref{Asubsec:UPDATE} zu finden. Die Einschränkung zu UPDATE ist im Kapitel \refTC{Einschränkung_CREATE_UPDATE} zu finden.

\section{DELETE}
\label{subsec:DELETE}
Das vereinfachte Vorgehen beim DELETE ist folgendermassen:
\begin{enumerate}
\item Überprüfe ob ein Filename mitgegeben ist. Wenn nicht, sende an Client, dass ein File ohne Namen nicht gelöscht werden kann.
\item Überprüfe im ersten SHM-Block, ob der Filename mit dem gesuchten Name übereinstimmt. Wenn ein Treffer vorhanden ist:
\begin{itemize}
\item setze den Write-Lock
\item setze den Filenamen = NULL
\item setzte den Filecontent = \grqq \textbackslash 0\grqq
\item setzte Flag im SHM-Block is free = TRUE
\item gib den Write-Lock wieder frei
\item destroy den rwlock mit pthread\_rwlock\_destroy(\dots)
\item sende dem Client die Nachricht, dass die Datei erfolreich gelöscht wurde.
\end{itemize}
\item Ist kein Treffer vorhanden, gehe zum nächsten Block und wiederhole dies so lange, bis ein Treffer vorhanden ist. Bei einem Treffer gehe zu Punkt 2
\item Ist bis zum letzten SHM-Block kein Treffer vorhanden, sende dem Client, dass der Filename nicht gefunden wurde und somit nicht gelöscht werden kann.
\end{enumerate}



\textbf{Beispiel für gültige Eingaben von DELETE:}\\
--DELETE filename.txt

Als Antwort des Servers gibt es zwei Varianten. Entweder \flqq File with the name xyz does not exist\frqq\ oder \flqq File with name xyz was successfully deleted.\frqq .

\textbf{Beispiele für ungültige Eingaben von READ:}\\
--DELETE

Wenn kein Filename mitgegeben wird, antwortet der Server mit der Mitteilung: \flqq Cannot delete a file without a name\frqq.

Der Programmcode für das Handling von DELETE ist im Listing \ref{listing:DELETE} in Anhang \ref{Asubsec:DELETE} zu finden.
